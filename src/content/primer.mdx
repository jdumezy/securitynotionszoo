## Overview

The security notions in this zoo are all stated in terms of a common vocabulary drawn from provable security.
This page introduces the key concepts needed to read and interpret these definitions: what kind of adversary we consider, how we measure advantage, and what the standard proof techniques look like.

## Probabilistic Polynomial-Time (PPT)

A **probabilistic polynomial-time** adversary is the standard model of an efficient attacker in modern cryptography.

- **Polynomial-time** means the adversary's running time is bounded by some polynomial in the security parameter $\lambda$ (e.g., the key length). This captures the intuition that the attacker is computationally bounded - it cannot, for instance, exhaustively try all $2^\lambda$ keys.
- **Probabilistic** means the adversary has access to a source of randomness and can make random choices during its execution. This models the fact that real attackers can flip coins, sample random values, and run randomized algorithms.

Formally, we write $\mathcal{A}(1^\lambda)$ for an adversary that takes the security parameter in unary as input. Security is required to hold for *all* PPT adversaries, which means no efficient algorithm can break the scheme, regardless of its strategy.

## Negligible Functions

When we say a scheme is secure, we do not require the adversary's success probability to be exactly $1/2$ (for indistinguishability) or exactly $0$ (for key recovery). Instead, we allow a small *advantage* - but it must be **negligible**.

A function $\varepsilon : \mathbb{N} \to \mathbb{R}_{\geq 0}$ is negligible if it vanishes faster than the inverse of any polynomial:

$$
\forall c > 0,\ \exists N,\ \forall \lambda > N : \varepsilon(\lambda) < \lambda^{-c}.
$$

Intuitively, a negligible quantity is so small that multiplying it by any polynomial in $\lambda$ still gives something that goes to zero. This means even if the adversary repeats its attack polynomially many times, its cumulative success probability remains negligible. Typical examples of negligible functions are $2^{-\lambda}$ and $\lambda^{-\log \lambda}$.

A function that is *not* negligible is called **non-negligible**. For example, $\lambda^{-3}$ is non-negligible: a scheme that can be broken with probability $\lambda^{-3}$ is considered insecure, because an attacker repeating the attack $\lambda^4$ times would succeed with overwhelming probability.

## Game-Based Security Definitions

Security notions are named **XXX-YYY**, where XXX denotes the security property the adversary tries to break (e.g., IND for indistinguishability) and YYY denotes the capabilities granted to the adversary (e.g., CPA, CCA1, CCA2). This naming convention is used throughout the zoo.

A **security game** (also called a security experiment) is a formal protocol between a **challenger** and an **adversary** $\mathcal{A}$. The challenger simulates the "honest" world - it generates keys, answers oracle queries, and issues challenges. The adversary tries to win the game by guessing a hidden bit, recovering a key, or producing a forgery.

The **advantage** of $\mathcal{A}$ in a game $G$ is a measure of how much better it performs than a trivial (random) strategy. Let $\Pr[\mathsf{Win}^G_\mathcal{A}]$ be the probability that $\mathcal{A}$ wins game $G$:

$$
\mathsf{Adv}^G_{\mathcal{A}}(\lambda) = \left| \Pr[\mathsf{Win}^G_\mathcal{A}] - \Pr[\text{random guess wins}] \right|.
$$

- For **search games** (like key recovery or forgery), the baseline is typically $0$, so $\mathsf{Adv} = \Pr[\mathsf{Win}]$.
- For **indistinguishability games** (like CPA or CCA), the baseline is $1/2$, so $\mathsf{Adv} = |\Pr[b' = b] - 1/2|$.

A scheme is **secure** with respect to game $G$ if $\mathsf{Adv}^G_\mathcal{A}(\lambda)$ is negligible for every PPT adversary $\mathcal{A}$.

## Reductions

A **reduction** is the primary tool for proving security. To show that scheme $\Pi$ is secure under assumption $\mathsf{A}$ (e.g., the hardness of LWE), we construct an algorithm $\mathcal{B}$ that:

1. receives an instance of problem $\mathsf{A}$ as input,
2. simulates the security game for $\Pi$ toward $\mathcal{A}$,
3. uses $\mathcal{A}$'s output to solve the $\mathsf{A}$ instance.

If $\mathcal{A}$ breaks $\Pi$ with non-negligible advantage, then $\mathcal{B}$ solves $\mathsf{A}$ with non-negligible probability - contradicting the assumption that $\mathsf{A}$ is hard. Therefore, no PPT adversary can break $\Pi$ with non-negligible advantage.

The quality of a reduction is measured by how much it degrades the advantage: if $\mathcal{A}$ wins with advantage $\varepsilon$, does $\mathcal{B}$ solve $\mathsf{A}$ with advantage $\varepsilon$, or only $\varepsilon / q$ where $q$ is the number of queries? **Tight reductions** (where the loss factor is constant) are preferred because they allow smaller security parameters without sacrificing concrete security.

## Falsifiable and Non-Falsifiable Assumptions

A reduction is only as meaningful as the assumption it reduces to. In the sense of [Naor (2003)](https://doi.org/10.1007/978-3-540-45146-4_6), an assumption is **falsifiable** if a claimed break can be demonstrated in an efficient challenge experiment: the challenger runs in polynomial time and can check whether the adversary has won. Standard search assumptions, decisional assumptions, and game-based notions such as IND-CPA or EUF-CMA fit this pattern.

A **non-falsifiable** assumption is not captured by such an efficiently checkable game. Typical examples include **knowledge assumptions**, which assert that an adversary that outputs some value must “know” a corresponding witness, and proofs in idealized models such as the random oracle, ideal cipher, or generic group model.

Plaintext awareness should be classified carefully. Some formulations behave like knowledge assumptions, because they posit an extractor for the plaintext “known” by the ciphertext creator. Other formulations, such as the PA0/PA1/PA2 notions of Bellare and Palacio, are better viewed as strong security notions for public-key encryption rather than assumptions in their own right.

Non-falsifiable assumptions and idealized models are not automatically invalid, but they are less direct evidence of concrete security. In the zoo, reductions are stated relative to standard falsifiable assumptions whenever possible, and uses of knowledge assumptions, idealized models, or circular/KDM-style assumptions are flagged explicitly.
## Game Hopping

**Game hopping** (also called game playing) is a proof technique introduced by [Bellare and Rogaway (2004)](https://eprint.iacr.org/2004/331) and [Shoup (2004)](https://eprint.iacr.org/2004/332) in which a proof proceeds through a sequence of games $G_0, G_1, \ldots, G_n$:

- $G_0$ is the original security game.
- $G_n$ is a trivial game in which the adversary's advantage is clearly $0$ (e.g., because its winning probability is exactly $1/2$ for an indistinguishability game).
- Each consecutive pair $G_i, G_{i+1}$ differs by a single conceptual step, and the change in the adversary's advantage between consecutive games is bounded.

By a triangle inequality argument:

$$
\mathsf{Adv}^{G_0}_\mathcal{A} \leq \sum_{i=0}^{n-1} \left| \mathsf{Adv}^{G_i}_\mathcal{A} - \mathsf{Adv}^{G_{i+1}}_\mathcal{A} \right| + \mathsf{Adv}^{G_n}_\mathcal{A}.
$$

Each hop is justified either by arguing the games are perfectly indistinguishable (so the adversary cannot tell the difference and has identical advantage), or by a reduction showing that distinguishing the two games would solve a hard problem.

## Hybrid Arguments

A **hybrid argument** is a special case of game hopping used to handle multiple independent instances of the same operation (e.g., handling multiple challenge ciphertexts).

Consider proving that a scheme secure against a single challenge is also secure against $q$ challenges.
Let $b \in \{0,1\}$ be the challenger's hidden bit.
In the multi-challenge game, the adversary can submit $q$ pairs of messages $(m_{j,0}, m_{j,1})$ and receive encryptions of $m_{j,b}$.
We define a sequence of hybrid games $H_0, H_1, \ldots, H_q$. In game $H_i$, the challenger encrypts the "right" message $m_{j,1}$ for the first $i$ queries, and the "left" message $m_{j,0}$ for the remaining $q - i$ queries.

Notice that $H_0$ corresponds to encrypting all "left" messages ($b=0$), and $H_q$ corresponds to encrypting all "right" messages ($b=1$).
Furthermore, consecutive hybrids $H_i$ and $H_{i+1}$ differ *only* in the encryption of the $(i{+}1)$-th ciphertext.
Any adversary that can distinguish $H_i$ from $H_{i+1}$ can be used to break the single-challenge security of the scheme.
By a triangle inequality (union bound) argument over the hybrids:

$$
\mathsf{Adv}^{\text{multi}}_\mathcal{A} \leq q \cdot \mathsf{Adv}^{\text{single}}_\mathcal{B}.
$$

If $q$ is polynomial in $\lambda$ and the single-challenge advantage is negligible, the total multi-challenge advantage remains negligible.
This is the standard argument establishing the equivalence of the single-challenge (Find-Then-Guess, FTG) and multi-challenge (Left-or-Right, LOR) formulations for standard CPA and CCA security.

Notably, for some FHE-specific notions (such as CPAD and vCCAD in the approximate setting), this hybrid argument *fails*.
The simulated games are not perfectly indistinguishable across hybrids because the decryption of approximate ciphertexts leaks varying amounts of noise depending on the exact queries.
This is precisely why the multi-challenge variants of those notions are strictly stronger than their single-challenge counterparts.

## Circular Security

Standard security definitions (like IND-CPA) guarantee that ciphertexts hide the underlying plaintext, provided the adversary's chosen messages *do not depend on the secret key*.
**Circular security** (also called **Key-Dependent Message** or KDM security) is the property that a scheme remains secure even when the adversary is explicitly given encryptions of the secret key or functions of the secret key under its corresponding public key.

Formally, an $\ell$-cycle arises when there exist key pairs $(pk_1, sk_1), \ldots, (pk_\ell, sk_\ell)$ such that $sk_i$ is encrypted under $pk_{i+1}$ (with indices wrapping around modulo $\ell$). The most critical case in FHE is a 1-cycle, where $sk$ is encrypted under its own $pk$.

Circular security assumptions are ubiquitous and mandatory in FHE because fundamental homomorphic operations require publishing evaluation keys that consist of encrypted secret-key material:

- **Bootstrapping:** The operation that refreshes a noisy ciphertext to evaluate circuits of arbitrary depth. A bootstrapping key contains encryptions of the secret key bits under the same public key.
- **Relinearization:** Used after homomorphic multiplication to reduce the expanded ciphertext back to its original dimension. Relinearization keys encrypt the tensor product of the secret key with itself ($sk \otimes sk$).
- **Rotation (Galois) keys:** Used in schemes like BGV, BFV, and CKKS to perform cyclic permutations on plaintext slots. These keys encrypt a Galois automorphism applied to the secret key.

Without circular security, standard IND-CPA proofs provide absolutely *no guarantees* once these evaluation keys are published.
Circular security is a strictly stronger assumption than standard CPA security: [Black, Rogaway, and Shrimpton (2002)](https://eprint.iacr.org/2002/100) and subsequent works demonstrated artificial encryption schemes that are provably CPA-secure but trivially leak the secret key if it is encrypted under itself.
In practice, mainstream FHE schemes (BFV, BGV, CKKS, TFHE) are heuristically believed to be circularly secure, but no unconditional proof exists reducing this property to standard lattice assumptions like LWE.

## Further Reading

The game-playing technique was systematized by [Bellare and Rogaway (2004)](https://eprint.iacr.org/2004/331).
[Shoup (2004)](https://eprint.iacr.org/2004/332) presents a great introduction to game hopping proofs.
Hybrid arguments and their role in proving CPA security appear in most cryptography textbooks; see Katz and Lindell, *Introduction to Modern Cryptography* (3rd ed.), or Boneh and Shoup, *A Graduate Course in Applied Cryptography*.
For the failure of hybrid arguments in the approximate FHE setting, see [Brzuska et al. (CIC 2025)](https://eprint.iacr.org/2024/812).
For a comprehensive treatment of security notions for FHE - including definitions, relations, separations, and constructions - see the PhD thesis of [Renard (2025)](https://theses.hal.science/tel-05421880v1/file/157007_RENARD_2025_archivage.pdf).
