## Overview

Replayable Chosen Ciphertext Attack (RCCA) security, also known as IND-RCCA, is a strict relaxation of the CCA2 game. Intuitively, while the CCA2 decryption oracle refuses only the challenge ciphertext, the RCCA decryption oracle refuses all ciphertexts that decrypt to one of the plaintext values submitted in the challenge request. In particular, this decryption oracle refuses to decrypt all rerandomizations of the challenge ciphertext in the context of correct schemes.

In the FHE context, RCCA is incompatible with homomorphic evaluation. Indeed, given a challenge ciphertext $c^*$ encrypting $m_b$, an adversary can homomorphically compute, say, $c' \leftarrow \mathsf{Eval}(f, c^*)$ for a known function $f$ satisfying $f(m_0),f(m_1)\not\in\{m_0,m_1\}$, then query the decryption oracle on $c'$ to recover $f(m_b)$, which trivially distinguishes $m_0$ from $m_1$ if $f(m_0) \neq f(m_1)$.

## Formal Definition

The IND-RCCA security game proceeds as follows:

1. The challenger generates a key pair $(pk, sk) \leftarrow \mathsf{KeyGen}(1^\lambda)$ and gives $pk$ to $\mathcal{A}$.
2. **Phase 1 (pre-challenge):** $\mathcal{A}$ has access to a decryption oracle $\mathsf{Dec}(sk, \cdot)$ and may submit arbitrary ciphertexts for decryption.
3. $\mathcal{A}$ outputs two equal-length messages $(m_0, m_1)$.
4. The challenger samples $b \xleftarrow{\textdollar} \{0,1\}$, computes $c^* \leftarrow \mathsf{Enc}(pk, m_b)$, and sends $c^*$ to $\mathcal{A}$.
5. **Phase 2 (post-challenge):** $\mathcal{A}$ **retains** access to the decryption oracle, but when called on ciphertext $c$, it refuses to answer the request if $\mathsf{Dec}(sk, c)\in\{m_0,m_1\}$.
6. $\mathcal{A}$ outputs a guess $b'$.

The advantage is defined as:

$$
\mathsf{Adv}^{\mathsf{IND\text{-}RCCA}}_{\Pi, \mathcal{A}}(\lambda) = \left| \Pr[b' = b] - \frac{1}{2} \right|
$$

The scheme is IND-RCCA secure if this advantage is negligible for all PPT adversaries.

## Attacks & Relevance

RCCA security prevents an adversary from re-randomizing or re-encrypting a challenge ciphertext to circumvent the CCA2 restriction. While the CCA2 oracle only refuses the exact challenge ciphertext $c^*$, an adversary could re-randomize $c^*$ into a different ciphertext $c'$ with the same plaintext and submit $c'$ for decryption. The RCCA oracle blocks this by refusing all ciphertexts that decrypt to $m_0$ or $m_1$.

RCCA was introduced by Canetti, Krawczyk, and Nielsen as a notion that is strong enough for most practical applications of CCA2 security (key transport, hybrid encryption) while being more permissive toward benign ciphertext transformations such as re-randomization. It is considered by many to be the "right" security notion for encryption in practice.

For FHE, RCCA is incompatible with homomorphic evaluation: given $c^*$ encrypting $m_b$, the adversary can compute $c' = \mathsf{Eval}(f, c^*)$ for a function $f$ with $f(m_0) \neq f(m_1)$ and $f(m_b) \notin \{m_0, m_1\}$, then query the decryption oracle to learn $f(m_b)$ and distinguish.

## Achieving This Notion

RCCA security is achieved by any CCA2-secure scheme (the implication CCA2 $\Rightarrow$ RCCA is immediate since the CCA2 oracle is strictly more restrictive). Canetti, Krawczyk, and Nielsen also gave direct constructions based on the Cramer-Shoup framework.

For FHE, RCCA is not achievable - the same fundamental incompatibility as CCA2 applies, since the adversary can use homomorphic evaluation to produce ciphertexts with discriminating plaintexts that are not in $\{m_0, m_1\}$.

## Further Reading

The RCCA notion was introduced by [Canetti, Krawczyk, and Nielsen (CRYPTO 2003)](https://link.springer.com/chapter/10.1007/978-3-540-45146-4_33). It sits strictly between CCA2 and vCCAD in the hierarchy of security notions. For a comparison with FHE-specific notions, see [Brzuska et al. (CIC 2025)](https://eprint.iacr.org/2024/812).
